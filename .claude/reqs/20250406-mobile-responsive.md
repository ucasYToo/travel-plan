# PRD: 移动端响应式样式优化

> 模板来源：`requirement-workflow/templates/prd-full.md`  
> 研究方法：`requirement-workflow/SKILL.md#研究阶段`（代码库审阅）

## 1. 产品概述

### 1.1 背景
当前首尔旅行地图应用在桌面端表现良好，但在移动端的适配存在明显不足。用户在旅行期间主要使用手机查看行程，因此移动端的可用性直接影响实际使用体验。

### 1.2 目标
优化应用在手机端的响应式表现，确保用户在旅行途中能够方便地查看地图、切换日期、浏览地点详情和交通方案。

### 1.3 范围
- **包含**：侧边栏、地图图层、顶部操作控件、弹窗模态框的移动端响应式调整。
- **不包含**：新增功能（如新的交通数据源、新的地图交互模式）、后端 API 修改、数据文件结构变更。

## 2. 用户故事与验收标准

### US-1：作为旅行者，我可以在手机上展开/收起侧边栏，以便在地图和行程信息间自由切换
**验收标准**：
- [ ] 在手机屏幕宽度（<640px）下，侧边栏默认收起，不遮挡地图主视图。
- [ ] 点击左上角菜单按钮，侧边栏从左侧滑出，占据几乎全屏宽度（如 85vw 或 `w-full`），并带有半透明遮罩。
- [ ] 点击遮罩或再次点击菜单按钮，侧边栏收起。
- [ ] 在桌面端（>=640px），侧边栏保持左侧常驻展示，行为不变。
- [ ] 通过 `Sidebar.test.tsx` 或新增测试覆盖开闭状态。

### US-2：作为旅行者，我可以在手机上切换城市、清除路线、查看全景，不遗漏桌面端已有的功能
**验收标准**：
- [ ] 手机顶部操作区（右上角）需包含：城市选择器、"清除路线"、"查看全景"三个入口。
- [ ] 由于空间限制，可采用底部固定工具栏（Bottom Sheet / Fixed Bar）或右上角下拉菜单的形式收纳这些操作。
- [ ] 桌面端 Quick Actions 保持现有胶囊按钮布局，不受影响。

### US-3：作为旅行者，我可以在手机上 comfortably 查看地点详情和交通详情弹窗
**验收标准**：
- [ ] `TransportModal` 和 `LocationDetailModal` 在手机端的宽度不超过屏幕宽度的 95%，最大宽度保持 `max-w-md`。
- [ ] 弹窗内容垂直方向可滚动，不超出视口，顶部和底部保留安全边距。
- [ ] 弹窗关闭按钮的触控区域不小于 44x44px。
- [ ] 弹窗内文字大小在手机端保持可读（不出现过小或截断）。

### US-4：作为旅行者，我在手机上能够更精准地点击地图标记和路线标签
**验收标准**：
- [ ] 在触碰设备（`hover:none` 或 `@media (pointer: coarse)`）上，地点标记（Marker）的最小触控区域扩展至至少 36x36px（可适当增加透明热区，不改变视觉大小）。
- [ ] 路线标签（Transit Label）的触控区域同样扩展，避免误触地图底层。

## 3. 研究发现（代码库审阅摘要）

1. **Tailwind CDN + 自定义断点**：项目已使用 Tailwind CSS（CDN 引入），并在 `App.tsx`、`Sidebar.tsx` 中使用了 `sm:` 断点，说明响应式意识已存在，但未彻底覆盖全部移动端场景。
2. **侧边栏实现**：`Sidebar.tsx` 当前使用固定宽度 `w-80`（320px），在 375px 宽手机上会挤压地图可视区域。移动端已有 `isOpen` 状态控制，但缺少遮罩层（overlay）和全屏感。
3. **顶部控件重复**：`App.tsx` 中通过 `hidden sm:flex` 和 `sm:hidden` 将桌面/移动端控件分成了两段代码，导致移动端缺失了城市选择、清除路线、查看全景按钮。
4. **模态框实现**：`TransportModal.tsx` 与 `LocationDetailModal.tsx` 均使用 `max-w-md` 和 `max-h-[85vh]`，在超小屏幕上可能出现水平溢出或文字过小的问题。
5. **测试现状**：`Sidebar.test.tsx` 已存在，但 `App.tsx` 和 `MapView.tsx` 在覆盖率配置中被排除，需要补充响应式相关测试或通过新增 utils/hooks 测试来覆盖。

## 4. 功能需求

### FR-1 侧边栏响应式重构
- 在小屏幕下，侧边栏容器应改为：
  - 宽度：`w-full sm:w-80`（手机全屏，桌面 320px）。
  - 增加遮罩层：`<div className="fixed inset-0 bg-black/30 z-[5] sm:hidden" onClick={closeSidebar} />`。
  - 默认收起：`-translate-x-full`，打开时 `translate-x-0`。
- 保持现有滚动行为、酒店卡片、行程列表、图例结构不变。

### FR-2 移动端顶部/底部操作栏
- 方案建议：在手机端使用底部固定工具栏（`fixed bottom-4 left-4 right-4`），集中放置：
  - 城市下拉选择器
  - "清除路线" 按钮
  - "查看全景" 按钮
- 桌面端保持现有右上角胶囊按钮布局，不引入底部栏。
- 减少 `App.tsx` 中重复渲染逻辑，建议将控制按钮抽取为独立组件（如 `MapControls.tsx`）。

### FR-3 弹窗模态框适配
- 统一 `TransportModal` 和 `LocationDetailModal` 的外层容器：
  - 水平内边距从 `p-4` 改为 `px-4 py-6 sm:p-4`，为小屏幕底部安全区留出空间。
  - 最大高度确保考虑地址栏：`max-h-[80dvh] sm:max-h-[85vh]`。
  - 内容区域 `overflow-y-auto`。
- 地址复制按钮在超窄屏下保持 `shrink-0`，地址文字自动换行（已有 `break-all`，需确认效果）。

### FR-4 地图标记触控增强
- 在不改变视觉样式的前提下，通过 CSS 为 `.custom-marker` 和 `.route-label` 增加透明热区。
- 添加媒体查询：
  ```css
  @media (pointer: coarse) {
    .custom-marker::after,
    .route-label::after {
      content: '';
      position: absolute;
      inset: -6px;
    }
  }
  ```
- 该 CSS 可写入 `src/index.css` 或注入到 `index.html` 的 `<style>` 标签中（项目使用 CDN Tailwind，可能未使用独立 CSS 文件，视实际文件结构而定）。

### FR-5 测试覆盖
- 补充 `Sidebar.test.tsx`：验证移动端侧边栏的打开/关闭行为（遮罩层出现/消失）。
- 若 `App.tsx` 无法直接测试，至少为新增控件组件编写独立单元测试，覆盖按钮点击后 `activeDay` 重置和 `resetView` 递增。
- 测试覆盖率目标 80%+（排除数据文件和主入口）。

## 5. 非功能需求与约束

### 5.1 性能约束
- 不能引入大型 UI 库（如 Material-UI、Ant Design），保持项目轻量。
- 继续使用 Tailwind CSS CDN，不迁移到构建时 Tailwind。

### 5.2 兼容性约束
- 支持 iOS Safari 和 Android Chrome 最新两个主版本。
- 使用 `dvh` 单位时需提供 `vh` 回退（Tailwind v3.4+ 的 `max-h-[80dvh]` 已自带回退逻辑，可保守使用 `@supports` 或同时写 `max-h-[85vh]`）。

### 5.3 设计约束
- 保持现有配色（白底+蓝紫渐变）和圆角风格（rounded-full / rounded-xl）。
- 不能改变桌面端现有布局和交互。

### 5.4 假设
- 用户的手机屏幕宽度主要介于 320px - 430px 之间。
- 旅行途中网络环境允许加载 CDN 资源。

## 6. 红旗检查清单

- [ ] 没有硬编码断点魔法值（统一使用 Tailwind `sm:`）。
- [ ] 没有引入新的外部依赖。
- [ ] 没有破坏桌面端现有布局。
- [ ] 弹窗内容不过长导致无法滚动关闭。
- [ ] 测试覆盖率不低于 80%。

## 7. 成功指标

- 在 iPhone SE（375px 宽）模拟器下，侧边栏可全屏展开、所有按钮可点击、弹窗不溢出。
- `npm run test` 全部通过，且新增/修改代码覆盖率 >= 80%。
- `npm run build` 成功，单 HTML 文件大小增长不超过 15%。
